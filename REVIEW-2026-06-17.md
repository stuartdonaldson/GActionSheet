### B1 â€” `WebApp.js:331-333` â€” `verify_action_rows` route bypasses all auth
**Location:** dispatcher at `src/WebApp.js:331-333`; handler `_handleVerifyActionRows` at line 800 (docstring: `{ secret, action: 'verify_action_rows', docUrl }`).
**Description:** The `doPost` dispatcher checks `payload.action === 'verify_action_rows'` unconditionally in the testToken-gated block (lines 311-339), with no `payload.testToken &&` guard â€” unlike its siblings `patch_action_status`/`delete_action_row` at lines 343/346, which do check â€” and no internal `_checkTestToken` call inside the handler itself. Reachable by anyone with the WebApp URL, no secret/token required; returns full ActionSheet rows (action text, assignee emails, status) for a given doc. An identical second branch at line 386-387, inside the `WEBAPP_SECRET`-gated block, is dead code â€” unreachable because the first branch always matches first.
**Why it matters:** Unauthenticated information disclosure of action item content/assignee PII. The dead duplicate branch also masks the *intended* secret-gated production route â€” a regression in that gate would never be caught by any test, since the unauthenticated path is always taken first.
**Recommended fix:** Add `payload.testToken &&` to the line-331 dispatcher condition (matching the `patch_action_status`/`delete_action_row` pattern), or call `_checkTestToken` inside `_handleVerifyActionRows`; then remove the now-dead duplicate branch at 386-387.
**Related:** S4 (the existing tests for this route can't detect the gap â€” see below). `docs/security-architecture.md` finding F3 already documents this exact gap; it has not been closed in code.
**Status: OPEN â€” code fix, not made in this pass.**

### S1 â€” `SyncManager.js:289` â€” `ReferenceError` in `syncAll()`'s own error-diagnostic path
**Location:** `src/SyncManager.js:289`.
**Description:** Inside `syncAll()`, the `if (!actionsSheet)` branch logs `GasLogger.log('sync.all.error', { msg: 'Actions sheet not found', eu: _syncEu, au: _syncAu })`. `_syncEu`/`_syncAu` are never declared anywhere in the file â€” the correct identity variable is `_syncId`, declared two lines above, with fields `_syncId.eu`/`_syncId.au`. **Why it matters:** This is the exact path meant to report a misconfigured/missing Actions sheet â€” the most actionable diagnostic in `syncAll()` â€” but it throws before the log call completes, so the outer `catch` (line 386) logs only the `ReferenceError` text, losing the real diagnostic. **Status: OPEN â€” code fix, not made in this pass.**

### S2 â€” `webSurvey.js` survey routes are entirely unauthenticated
**Location:** `src/webSurvey.js:40-71`, dispatched from `src/WebApp.js:75-77`.
**Description:** `cmd=survey&action=add|submit` writes to the sheet with no secret/token check of any kind. `_findSurveyNameColumn` matches respondent by name case-insensitively with no ownership check, so a second submission under the same name silently overwrites the first respondent's ranking/comment.
**Why it matters:** Public endpoint, no rate limiting or auth â€” spam/vandalism of survey data and respondent impersonation are both possible (anyone can submit as "Jane Doe" and overwrite Jane's real ranking).
**Recommended fix:** Either add a simple per-respondent claim token, or â€” if intentionally public-facing (plausible for a congregation survey) â€” explicitly document the trust model. This is a design decision, not a one-line fix.
**Related:** B3 (same file, coverage gap).
**âš  Downgraded 2026-06-17 (post-report):** per B3's note, `webSurvey.js` is a draft experiment slated for removal before release. No auth-hardening work is warranted on code that won't ship â€” the only actionable item is tracking its removal.
**Status: OPEN, but moot â€” no fix needed if the file is removed before release. See "Survey code disposition" note at the end of this report.**

### S3 â€” `WorkspaceAddonCard.js:888-897` â€” `onDeleteAction` lacks error handling its sibling has
**Location:** `src/WorkspaceAddonCard.js:888-897`, compared to `onSetActionStatus` at lines 851-882.
**Description:** `onSetActionStatus` wraps its body in try/catch and returns a CardService error notification on failure. `onDeleteAction` calls `sidebarDeleteAction(globalId)` and `buildHomepageCard(...)` with no try/catch â€” any exception (e.g. `DocumentApp.openById` failing, or the delete handler's `UrlFetchApp.fetch` throwing) propagates uncaught to the CardService runtime as a generic error instead of the project's normal "Delete failed: ..." notification.

**Recommended fix:** Wrap `onDeleteAction`'s body in the same try/catch + `CardService.newNotification().setText('Delete failed: ' + err.message)` pattern as `onSetActionStatus`.
**Status: OPEN â€” code fix, not made in this pass.**
