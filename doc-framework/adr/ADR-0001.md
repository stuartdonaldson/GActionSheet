# ADR-0001: Documentation Framework Selection

Status: Accepted
Date: 2026-03-02

## Context
This project (and others in the same tooling family) needed a documentation
framework that would:
- Support fast developer context reload after gaps between sessions
- Work effectively with AI coding agents (Claude Code, GitHub Copilot)
- Scale from solo CLI tools to multi-stakeholder systems without requiring
  a different vocabulary at each scale
- Align with established industry standards without imposing unnecessary ceremony
- Support agile development — living documents, decision tracking, minimal
  maintenance overhead

Frameworks considered:
- arc42 — comprehensive, well-established, 12-section template. Strong on
  architecture documentation. No concept of living state documents, backlog,
  or AI optimization. Heavyweight for small projects.
- C4 Model — diagrams-first, minimal prose. Good for architecture visualization
  but not a complete documentation system.
- Diátaxis — strong for user-facing documentation. Not suited to internal
  architecture and decision documentation.
- Custom minimal approach — no structure, document as needed. Low overhead
  but inconsistent across projects and poor AI retrieval.

## Decision
Adopt a custom three-tier framework (Minimal / Standard / Extended) that:
- Uses arc42 section terminology for familiarity and transferability
- Adds living documents (PLAN.md, BACKLOG.md) not present in arc42
- Adds AI-specific routing and maintenance via CLAUDE.md
- Adopts ADR format (Nygard) for decision tracking
- Adopts Mermaid as the default diagramming standard
- Scales by activating document types as complexity warrants, rather than
  imposing the full structure on every project
- Separates shared standards from tier-specific prompts so rules are
  maintained in one place

The active minimum for any project is: CONTEXT.md (or README.md §CONTEXT),
PLAN.md, BACKLOG.md, ADRs, and CLAUDE.md. Additional documents activate
when the gap is felt, not in anticipation of it.

## Consequences
- Developers familiar with arc42 will recognize section names and concepts
- Projects at different scales use consistent terminology, making graduation
  between tiers straightforward
- AI agents have explicit routing and maintenance instructions, reducing
  session startup overhead
- The framework itself requires maintenance as it evolves — stored in
  /doc-framework/ in each repo, sourced from a central tooling location
- Disposition planning (Phase 1.5) adds overhead during migration of existing
  projects but is unnecessary for greenfield projects
- Quality scenarios and stakeholder documentation (Extended tier) add
  overhead only when multiple stakeholder groups or compliance requirements
  exist
